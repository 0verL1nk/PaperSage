# 文献助手升级调研报告（简历导向）

## 1）当前产品差距总结（为什么用户会觉得像“玩具”）

### 现有代码中的核心痛点

1. **流水线可靠性偏弱**
   - 旧/新数据库 schema 混用，导致 Python 运行时需要补丁式兼容。
   - 异步任务路径会退化为同步路径，用户心智模型不清晰。
   - 证据：`utils/task_queue.py`、`utils/page_helpers.py`、`utils/utils.py`。

2. **状态与 UX 一致性脆弱**
   - 各页面独立管理状态、重试和 rerun 循环。
   - 高频 rerender 导致视觉抖动（特别是问答流式输出）。
   - 证据：`pages/1_🤓_原文提取.py`、`pages/2_😶‍🌫️_论文总结.py`、`pages/4_🤖_论文问答.py`。

3. **RAG 价值受限于检索质量与持久化策略**
   - 内存索引生命周期短，引用/校验体验弱。
   - 查询重写、重排、评测等检索质量控制尚未产品化。
   - 证据：`utils/local_rag.py`、`utils/agent_capabilities.py`。

4. **Agent 能力不完整，尚未达到产品级**
   - 已有 tool calling，但缺少显式 planner/executor 分层。
   - skill 目前更像本地模板映射，不是受治理的运行时能力。
   - MCP 尚未接入。

5. **简历信号目前偏弱**
   - 缺少高质量项目常见的基准评测、质量仪表盘与架构叙事。

### 1.1 当前功能是否命中科研用户痛点（基于现状功能盘点）

| 科研用户高频任务/痛点 | 当前已有功能 | 命中度 | 主要缺口 |
|---|---|---|---|
| 快速读懂单篇论文（时间不足） | 原文提取、论文总结、问答 | 中-高 | 缺少“结论可靠性”提示与证据链可视化 |
| 多篇文献横向比较（综述/开题常见） | 基础问答与总结 | 低 | 缺少跨文献对齐、对比表自动生成、冲突观点聚类 |
| 引用可信与可追溯（写作与答辩刚需） | 局部引用能力（RAG 基础） | 低-中 | 缺少引用覆盖率、证据定位、一键核验流程 |
| 长周期研究连续性（多轮会话） | 基础会话态 | 低 | 缺少长期记忆、研究会话资产沉淀（问题-结论-证据） |
| 结果可复现（科研规范） | 任务日志初步可见 | 低-中 | 缺少固定评测集、版本化 prompt/配置、实验报告导出 |
| 工作流效率（从阅读到写作） | 改写、脑图、问答等分散页 | 中 | 缺少统一研究会话编排，页面间上下文割裂 |
| 成本与隐私平衡 | 本地 RAG + 可配置云端调用 | 中 | 缺少显式路由策略与可解释决策（为何本地/为何云端） |
| 失败恢复与稳定性 | 已有重试和队列改进 | 中 | 缺少可视化重试策略、失败分级与自动降级策略 |

结论（现状判断）：
- 当前实现对“单篇快速理解”已经有可用价值。
- 对“科研生产级场景”（多文献比较、可追溯引用、可复现评测、长期研究记忆）仍明显不足。
- 用户反馈“功能比较简陋”的核心原因，不在于功能数量，而在于缺少端到端研究闭环与质量证明机制。

### 1.2 科研人群核心痛点画像（按任务链路）

1. **检索与筛选成本高**
   - 关键词不稳定、结果噪声大、跨库检索重复劳动多。

2. **阅读吞吐不足**
   - 需要在有限时间内处理大量论文，且常需在“细读”和“速读”之间切换。

3. **跨文献综合难**
   - 真正有价值的是“比较、归纳、冲突定位、研究空白识别”，不是单篇摘要。

4. **证据可信与引用可追溯压力大**
   - 科研写作和答辩场景需要可定位证据，不能只给结论。

5. **写作与阅读割裂**
   - 从阅读笔记到论文段落重写、引用格式整理，链路常被手工工具打断。

6. **长期项目记忆缺失**
   - 研究周期长，历史提问、关键结论、反证线索难沉淀为“可复用资产”。

7. **可复现与可审计不足**
   - 若不能复盘“当时如何得到这个结论”，就难以用于正式学术产出。

8. **多语言阅读门槛**
   - 非母语论文在术语密度、句法复杂度上造成显著理解负担。

9. **工具切换成本高**
   - PDF 阅读器、笔记、引用管理、AI 工具分散，状态难同步。

10. **对 AI 的信任门槛高**
    - 用户担心幻觉、过度概括、引用错位，导致“看起来快，实则不敢用”。

### 1.3 当前版本“简陋感”的具体来源

- **功能是“点状能力”，不是“闭环能力”**：提取、总结、问答、改写、脑图都可用，但尚未串成一个研究会话。
- **质量证据不足**：缺少 answer faithfulness、citation coverage、retrieval hit 这类可量化质量指标。
- **过程不可解释**：用户难以看到“为什么调用该工具、证据来自哪里、置信度为何如此”。
- **长期价值弱**：缺少可沉淀、可回放、可复用的研究记忆与实验记录。

### 1.4 面向科研用户的最小产品升级标准（MVP+）

满足以下 6 条，产品感会显著提升：

1. 单一 `Research Session` 工作台（替代分散页面）。
2. Plan-Act-Verify 可视化执行链路（含每步输入/输出与耗时）。
3. 每个结论都能回链到证据片段（可点击定位来源段落）。
4. 固定评测集与质量分（最少：faithfulness、relevance、citation coverage）。
5. 研究记忆沉淀（问题、结论、证据、反例、未决事项）。
6. 本地/云路由策略可解释（为何走本地、为何升级云端模型）。

## 2）2025-2026 Agent/AI 技术调研与项目相关性

### A. MCP 生态（高战略价值）
- MCP 统一了模型到工具/数据的访问标准，降低定制连接器成本。
- 相关性：与本项目接入文献源、引用库、本地索引器和 Web 工具的需求直接匹配。
- 依据：Anthropic MCP 公告与 MCP 官方文档（`modelcontextprotocol.io`、Anthropic News）。

### B. Agent 架构趋势：可组合工作流优于“黑盒自治”
- 生产实践倾向先做显式工作流（路由、并行、orchestrator-workers、evaluator-optimizer），再提升自治。
- 相关性：本项目应演进到可控的 Plan-Act + Eval 闭环，提升可调试性与可信度。
- 依据：Anthropic《Building effective agents》。

### C. LangGraph 风格的可控编排
- 强调有状态长流程、中断/人工介入、过程流式、记忆机制。
- 相关性：适合论文多步骤流程（导入 -> 检索 -> 分析 -> 验证 -> 产出）。
- 依据：LangGraph 产品/文档方向。

### D. RAG 工程趋势：评测优先与可溯源可靠性
- 行业已从“能检索就行”转向“可量化 groundedness（忠实度、引用准确率、命中质量）”。
- 相关性：可通过“可验证答案 + 引用置信分”形成差异化。

### E. 本地/云混合模型服务
- 隐私敏感的检索/向量化放本地；高质量复杂生成按策略走云端。
- 相关性：与当前本地 embedding 方向一致，应升级为显式策略引擎。

### F. 实战模式矩阵（可落地优先级）

| 模式 | 价值 | 复杂度 | 风险 | 简历区分度 |
|---|---|---|---|---|
| 带工具预算护栏的 ReAct loop | 适合开放式论文问答与迭代检索的稳定基线 | M | M | 4/5 |
| Plan-Act(-Verify) 状态机 | 提升多步骤研究流程可靠性，执行过程可审计 | M | M | 5/5 |
| MCP 适配层（先只读） | 标准化工具/数据接入，降低集成债务 | M | M | 5/5 |
| 受治理工具注册表（allowlist/timeout/retry/redaction） | 防止失控调用，强化生产稳定性 | S-M | L-M | 4/5 |
| 会话 + 情节记忆 | 强化长时阅读会话与追问连续性 | M | M | 4/5 |
| 检索质量栈（重写 + 混合检索 + 重排 + 引用） | 对答案质量与可信度提升最直接 | M | M | 5/5 |
| 评测/可观测性流水线（trace + groundedness 指标） | 支撑可量化迭代与作品级工程证据 | M | L-M | 5/5 |
| 本地/云混合路由策略 | 平衡隐私、成本、时延，便于渐进扩展 | M | M | 5/5 |

本仓库落地建议：
- 先在现有 Streamlit 流程接入 Plan-Act-Verify 与工具治理。
- MCP 先做 1-2 个只读连接器。
- 在提升 agent 自治前，优先补齐 retrieval/eval。

## 4）建议的差异化产品方向

### 主题：“可信文献研究 Agent”（Trustworthy Research Agent）

建议从“多功能分页工具”转为**单一编排式研究工作流**，并设置明确检查点：

1. **Plan**：解析研究意图并生成步骤计划。
2. **Act**：执行检索/工具调用（文档、Web、引用源）。
3. **Verify**：进行 groundedness 校验、引用一致性检查与置信评分。
4. **Deliver**：输出总结、问答、脑图、改写等可关联产物。

可写进简历的关键差异点：
- 可解释执行轨迹（调用了什么工具、为何调用、结果质量）。
- 有据可查的答案质量评分与引用审计。
- 成本/隐私感知的本地-云混合策略引擎。
- 对高风险或高成本步骤支持人工审批（HITL）。

## 5）8 周升级路线图（小步快跑但有影响力）

### Phase 1（第 1-2 周）：架构基线
- 引入工作流抽象（Plan -> Act -> Verify）。
- 统一 task/event schema 与 tracing ID。
- 收口 retry/backoff/cancel 语义。

### Phase 2（第 3-4 周）：补齐 Agent 能力
- 增加显式 planner 模块。
- 增加受治理的工具注册表（allow/deny、timeout、retry、redaction）。
- 增加 MCP 适配层（先接 1-2 个 MCP endpoint）。

### Phase 3（第 5-6 周）：RAG 质量与评测
- 增加检索诊断（top-k recall 代理指标、chunk 命中统计）。
- 增加答案 groundedness 检查与引用覆盖率评分。
- 基于代表性论文问题构建离线评测集。

### Phase 4（第 7-8 周）：产品打磨 + 简历产出
- 增加可观测性面板（延迟、工具成功率、grounding 分数）。
- 增加可复现实验报告与演示脚本。
- 提供一键本地运行配置与确定性测试数据集。

## 6）简历写法建议（高信号）

1. 构建了**有状态研究 Agent**，具备显式 Plan-Act-Verify 闭环与工具治理。
2. 实现了**本地/云混合 AI 策略**，兼顾隐私保护、成本控制与质量。
3. 搭建了**grounding/引用评测流水线**与质量可观测面板。
4. 集成了**基于 MCP 的可扩展工具生态**，实现标准化数据/工具接入。
5. 通过 schema 迁移、重试策略与类型契约提升整体稳定性。

## 7）风险与缓解

- **风险**：在基线稳定前过度引入复杂编排。  
  **缓解**：第一阶段只做一个工作流 + 三个核心工具。

- **风险**：MCP 接入面过大导致实现失控。  
  **缓解**：先做只读 MCP 连接器，并启用严格 allowlist。

- **风险**：评测工作陷入“研究型无底洞”。  
  **缓解**：冻结 50 条固定 benchmark query，仅围绕该集合迭代。

## 8）建议立即执行的下一步

实现一个**单页“Research Session”**替代当前分散页面，包含：
- 显式计划展示，
- 工具执行时间线，
- grounded answer + citation score，
- 可导出的研究报告产物。

这是从“功能拼盘”最快升级到“可讲述、可面试、可复现”的 Agent 产品叙事路径。

## 9）补充上线清单（来自额外 2025-2026 扫描）

第二轮调研（`bg_70427dfa`）总体与上述方向一致，补充以下可直接落地建议：

1. **交互问答用 ReAct，批量研究任务用 Plan-Act**
   - 两种模式显式区分，聊天默认 ReAct。

2. **关键可信步骤加入 HITL 审批**
   - 对“高成本 Web 调用”“最终引用导出”等动作增加人工确认。

3. **先可观测，再自治**
   - 在提升 agent 复杂度前，先保证每步工具调用、延迟、重试、失败原因可追踪。

4. **CI 接入固定评测包**
   - 准备稳定评测数据集，对每次 agent/RAG 变更执行忠实度/相关性检查。

5. **混合路由策略配置化**
   - 简单检索/摘要优先本地模型，复杂综合推理再走云模型。

6. **MCP 渐进式接入**
   - 先将现有内部工具封装为 MCP 兼容接口，再扩展外部连接器。
