## 1. 目标与成功标准

### 1.1 产品目标

将系统从“单篇论文问答助手”迭代升级为**科研级文献工作台**，需具备以下能力：

1. 基于证据的回答，来源可追溯。
2. 更好的文献发现能力，补充缺失的上下文。
3. 持久化且可扩展的检索基础架构。
4. 结构化输出，用于对比与综合分析。

### 1.2 交付目标 (8周计划)

本计划结束时，系统应支持：

- 带有可检索证据卡片且有来源支撑的问答。
- 学术平台辅助的论文检索（生产环境至少接入一个平台）。
- 持久化向量索引选项（保留内存索引作为备用方案）。
- 基础的方法对比结构化输出与导出功能。

### 1.3 KPI 指标

- **证据可追溯率**: >= 80% 的助手回答包含至少 1 个结构化证据项。
- **中位数响应延迟**: 对于简短问答，相比当前基线延迟增加不超过 25%。
- **避免重复索引**: >= 70% 的重复查询命中持久化索引，而不是触发全面重建。
- **Top-5 相关性 (人工评估)**: 相比当前基线提升 10%。

## 2. 当前基线状况 (As-is)

基于代码库检查：

- UI 入口: `main.py` (聊天、工作流路由、思维导图渲染、归档展示)。
- 检索模块: `agent/local_rag.py` 结合 `InMemoryVectorStore` + FastEmbed + 可选的 FlashRank 重排。
- Agent 工具: `agent/capabilities.py` (`search_document`, `search_web`, `use_skill`)。
- 多 Agent 编排: `agent/multi_agent_a2a.py`, `agent/workflow_router.py`。
- 持久化: SQLite (`utils/utils.py`)，归档功能在 `agent/archive.py`。
- 测试: 存在于 `tests/unit` 和 `tests/integration`。

从产品角度看的主要差距：

1. `search_document` 返回纯文本，而非结构化的来源证据。
2. `search_web` 是通用搜索，未集成学术搜索提供商。
3. 向量存储优先使用内存；主流程中没有持久化索引策略。
4. 跨论文的结构化综合分析和引用工作流能力有限。

## 4. 详细迭代待办清单

## 阶段 A (第 1-2 周): 优先实现证据可追溯性

### A1. 检索契约重新设计 (必做)

**目标:** 将检索输出重新定义为结构化的证据列表。

**影响的文件:**
- `agent/local_rag.py`
- `agent/capabilities.py`
- `main.py`
- `tests/unit/test_local_rag_rerank.py` (扩展)
- 新增测试: `tests/unit/test_evidence_payload_schema.py`

**新证据数据结构 (契约):**

```json
{
  "doc_uid": "string",
  "chunk_id": "string",
  "text": "string",
  "score": 0.0,
  "source": "document",
  "page": 0,
  "offset_start": 0,
  "offset_end": 0
}
```

**实现任务:**
1. 在拆分/索引阶段保留块 (chunk) 元数据。
2. 从检索路径内部返回 `List[EvidenceItem]`。
3. 在工具输出中暴露 JSON 字符串，以保持向后兼容性。
4. 在 UI 中添加解析辅助函数，以渲染证据卡片并实现优雅降级。

**验收标准:**
- 问答响应路径在有证据时显示证据卡片。
- 当证据解析失败时，不中断现有的答案流。
- 单元测试覆盖空文档、未命中、格式错误的证据等情况。

### A2. 答案 + 证据展示 UX

**影响的文件:**
- `main.py`
- `utils/page_helpers.py` (可选的提取逻辑)

**任务:**
1. 为每条助手消息添加可折叠的证据面板。
2. 如果可用，显示分数/来源/页码。
3. 添加“复制引用片段”按钮 (先实现简单的文本格式)。

**验收标准:**
- 用户可以检查至少排名前 3 的证据片段。
- 渲染过程不阻塞流式输出。

## 5. 工程契约 (必须遵守)

### 5.1 检索契约

- **输入**: `(doc_uid, query, top_k)`
- **输出**: `List[EvidenceItem]`
- **错误模式**: 返回空列表 + 错误遥测，不要让聊天路径崩溃。

### 5.2 提供商契约

- **输入**: `(query, limit, timeout)`
- **输出**: `List[PaperMetadata]`
- **错误模式**: 具有回退决策的类型化失败 (`search_web` 或请求用户澄清)。

### 5.3 UI 渲染契约

- 流式解答为第一优先级。
- 结构化组件 (证据/表格/思维导图) 是附加的且非阻塞的。

## 6. 需覆盖的边缘用例

1. 空/极短的文档文本，以及 OCR 失败的文档。
2. 超长查询和中英文混合的多语言查询。
3. 提供商超时 / 触发速率限制 / 获取到部分元数据。
4. 重排器 (Reranker) 不可用 (依赖缺失) 时的回退行为。
5. 持久化索引损坏或版本不匹配。

## 9. 风险登记与缓解措施

1. **提供商不稳定 / API 限制**
   - 缓解: 超时控制 + 重试 + 备用提供商 + 回退到 `search_web`。

2. **Schema 漂移破坏 UI 解析**
   - 缓解: 版本化 schema + 具有容错默认值的解析器。

3. **持久化后端的运维复杂性**
   - 缓解: 特性开关逐步发布，本地用户默认保持内存模式。

4. **延迟退化**
   - 缓解: 限制 `top_k`，缓存嵌入向量，重型路径采用异步操作。

5. **版权/访问限制**
   - 缓解: 元数据优先流程，仅对 OA 链接启用自动获取，用户确认后再导入。

## 10. 发布与回滚计划

### 发布 (Rollout)

1. 为新检索/提供商/后端发布隐藏开关。
2. 首先对内部/测试用户启用。
3. 每个里程碑后观察日志和测试指标 2-3 天。
4. 逐步向所有用户开放。

### 回滚 (Rollback)

- 关闭新开关以恢复之前的行为：
  - 关闭结构化证据 -> 恢复纯文本检索路径。
  - 关闭学术提供商 -> 恢复现有网络搜索路径。
  - 关闭持久化后端 -> 恢复内存后端。
