# 2026-03-08 项目架构治理与重构计划

## 1. 背景与问题定义

当前仓库存在“新架构与遗留实现并存”的状态，导致维护成本高、定位问题慢、迭代风险大。核心问题：

1. `utils/utils.py` 体量过大（3000+ 行），混合数据库迁移、业务逻辑、LLM 调用、页面状态相关逻辑。
2. `agent/adapters/*` 仍大量透传 `utils.utils`，形成分层反向依赖，边界不清晰。
3. Streamlit 多页面存在重复初始化逻辑（DB 初始化、用户初始化、session_state 默认值）。
4. `ui/agent_center_page.py` 单文件职责过重，编排、状态处理、渲染混在一起。
5. Worker (`utils/tasks.py`) 依赖 `sys.path.insert` 与兜底导入，部署稳定性和可维护性较差。

## 2. 重构目标（Goal）

1. 明确分层边界：`UI -> Application -> Domain -> Adapters -> Infra/Repository`。
2. 将遗留 `utils.utils` 按职责拆分，降低单模块复杂度。
3. 页面初始化与会话初始化标准化，减少重复代码与行为偏差。
4. 让质量门禁覆盖范围逐步扩大，避免“核心模块可控、边缘模块失控”。
5. 在不破坏现有功能前提下逐步迁移，保证可回滚。

## 3. 非目标（Non-Goal）

1. 不进行 UI 风格重做。
2. 不一次性重写全部 Agent 流程。
3. 不在本轮将 SQLite 切换为其他数据库。

## 6. 风险与回滚策略

风险：
1. 迁移期间出现函数行为不一致导致回归。
2. 页面状态 key 变更导致历史会话体验异常。
3. worker 导入路径调整影响生产环境启动。

回滚策略：
1. 每阶段独立 PR，避免大爆炸合并。
2. 保留兼容适配层（deprecated wrapper）1-2 个迭代周期。
3. 出现回归时按阶段回滚，不跨阶段混合回退。

## 7. 完成标准（Definition of Done）

1. `utils/utils.py` 明显瘦身，核心业务不再新增到该文件。
2. 跨层依赖方向清晰，无 `agent <-> utils` 双向耦合。
3. 页面初始化逻辑统一、重复代码显著减少。
4. Agent Center 核心入口职责清晰、可测性提升。
5. 质量门禁覆盖范围扩大并稳定运行。
